Read `../00-master-plan.md` В§1 (invariants), В§2 (why F0 freezes everything), В§4 (the contracts you author),

## Goal

Stand up a **runnable, tested spine skeleton** that reduces two append-only logs + a live config store to a
JSON view via a pure `rebuild()`, with every downstream contract frozen and every abstraction invariant
enforced by a CI gate. After F0 merges, Wave-1 sessions implement pure stages in their own dirs against these
contracts + fixtures вЂ” never against each other.

## Scope (build these)

1. **Scaffold** вЂ” `pyproject.toml` declaring the **full locked dependency set** (master В§4.1: pydantic,
   networkx, anthropic, google-genai[opt], fastapi, uvicorn, pyyaml, python-dateutil, rapidfuzz, rank-bm25,
   geopy, pytest, pytest-cov, ruff, mypy, httpx, respx). Package `backend/chanakya/`; scaffold the **`backend/`** tree (all Python code/tests/deps live there вЂ” master
   В§4.1 rooting convention). `ruff` + `mypy` config.
   `Makefile` skeleton with stubbed targets (`extract build ingest ask run test lint` в†’ echo TODO).
2. **Schemas** (`chanakya/schemas/`) вЂ” pydantic models for every record in master В§4.2: `ClaimRecord`,
   `SourceRegistryEntry`, `DecisionRecord`, `DerivedAssertionState`, `KnownGap`, `Alert`, the **value objects `Date`/`Location`/`Quantity`** (shapes + canonical slots only вЂ” the
   normalization *adapters* are INGEST's, and F0 puts **no** network/parse call in an on-instantiation
   validator, per В§4.2 shape-vs-normalization), plus the config
   models (ontology/sources/credibility/resolution/templates/subjects/observables) and the view-JSON + API response
   models (master В§4.8, `product/03` AвЂ“H). Human-readable `claim_id` scheme (`d05-row12`).
3. **Config store** (`chanakya/config/`) вЂ” a loader that seeds a **live, writable** in-process config store
   from `config/*.yaml` and exposes read + write (hot-config); versioned. Wave-1 reads config through this,
   never from files directly.
4. **Store** (`chanakya/store/`) вЂ” SQLite **evidence log** + **decision log**, append-only (no UPDATE/DELETE
   in the API surface), with `append`, `query`, `seed_from(path)`, `replay`. Boot reads the baked baseline;
   runtime writes append to a container-local copy.
5. **`rebuild()` skeleton** (`chanakya/view/`) вЂ” the pure orchestrator calling the five stage interfaces in
   fixed order (master В§4.3), with **trivial stubs** shipped for each (identity/empty so the skeleton runs).
   Build **real** (not stubbed): supersedes-vs-contradicts on `resolved_ref`; `apply_lens(view, subject)`
   scoping (N-hops + materiality filter); JSON export matching В§4.8. Define each stage as an importable
   function with the frozen signature.
6. **Stub packages** вЂ” create `chanakya/{resolve,credibility,sufficiency,materiality,observe,agent,ingest,
   hitl,api}/__init__.py` each exporting its stage function as a trivial stub, so Wave-1 sessions only *edit*
   files inside their own dir (never create in a shared parent).
7. **Golden fixtures** (`tests/fixtures/`) вЂ” a small hand-authored evidence+decision log + expected view
   JSON, plus per-stage fixtures (resolution inputв†’partition; claim setв†’`assertion_confidence`;
   assertionв†’status/known-gap; two viewsв†’alert). Synthetic + stable; **decoupled from the C corpus**.
8. **Gate suite** (`tests/gates/`) вЂ” implement **G1вЂ“G12** (master В§5) with seed fixtures. These run in CI on
   every PR; they are the enforcement of the abstraction design.
9. **CI** (`.github/workflows/`) вЂ” `ruff` + `mypy` + `pytest` (incl. gates) on push/PR.
10. **`PROGRESS.md` seed** (repo root) + copy the handoff template into place.

## Acceptance criteria
- [ ] `make test` green: `ruff`, `mypy`, `pytest` incl. all **G1вЂ“G12** gates.
- [ ] `rebuild()` runs end-to-end over the golden fixtures with stub stages and emits view JSON matching the
      В§4.8 shapes; **G2 determinism** (byte-identical across 2 runs) and **G1 purity** (network/LLM/clock/RNG
      patched to raise) both pass.
- [ ] Every stage interface is importable with its frozen signature; each stub is trivially correct.
- [ ] Store rejects UPDATE/DELETE (G3); retraction-as-appended-claim round-trips; `seed_from`/`replay` are
      deterministic.
- [ ] `apply_lens` filters the fixture graph to N-hops-from-anchor + materiality; supersede/contradict
      fixture behaves per master В§4.3.
- [ ] `config` store round-trips a hot write and `rebuild()` reflects it (no restart).
- [ ] Fixtures + gates are documented enough that a Wave-1 session can extend the relevant gate without
      reverse-engineering.
